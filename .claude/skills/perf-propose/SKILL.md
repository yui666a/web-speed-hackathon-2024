# パフォーマンス改善施策提案スキル

## 概要

perf-measure の計測結果を入力として、スコアリング配点を考慮した改善施策を提案する。
机上の提案ではなく、コードベースを実際に調査して具体的な変更内容まで落とし込む。

## 前提条件

- `perf-measurements/latest/report.md` が存在すること（perf-measure で計測済み）
- 存在しない場合は「先に perf-measure で計測してください」と案内する

## ワークフロー

### Step 1: 計測結果の読み込み

1. `perf-measurements/latest/report.md` を読んで現状スコアを把握
2. 各ページの Lighthouse JSON（`home.json`, `author.json`, `book.json`, `episode.json`）から詳細データを取得
3. 失点分析テーブルから、改善インパクトの大きいメトリクスを特定

### Step 2: スコアリング配点に基づく優先度判定

採点の配点ウェイトを常に意識して優先度をつける：

| メトリクス | 配点（各ページ） | 改善の主な手段 |
|-----------|----------------|--------------|
| **TBT** | **30点** | JS バンドル削減、コード分割、不要ライブラリ除去、メインスレッド負荷軽減 |
| **LCP** | **25点** | 画像最適化、プリロード、SSR 改善、クリティカルパス短縮 |
| **CLS** | **25点** | 画像/フォントのサイズ明示、レイアウトシフト防止 |
| **FCP** | **10点** | クリティカル CSS インライン化、レンダーブロック除去 |
| **SI** | **10点** | 上記改善で連動して改善される |

**優先順位の考え方**: 「失点 × 改善の実現可能性」で判断する。
失点が大きくても実装が極めて困難なものより、中程度の失点でも簡単に直せるものを先に提案する。

### Step 3: コードベースの調査

提案を具体的にするため、実際のコードを調査する。以下を重点的に確認：

#### TBT が高い場合
- `workspaces/client/tsup.config.ts` - バンドル設定（ミニファイ、ツリーシェイキング）
- `workspaces/client/src/index.tsx` - エントリポイントの import
- `workspaces/app/package.json` - 不要な依存関係
- polyfill の使用状況（core-js, es5/6/7-shim, regenerator-runtime）
- 重いライブラリ（three.js, canvaskit-wasm, jQuery, lodash, moment-timezone）

#### LCP が高い場合
- `workspaces/server/src/routes/image/` - 画像配信の最適化状況
- `workspaces/app/src/foundation/components/Image/` - 画像コンポーネントの実装
- SSR の実装状況（`workspaces/server/src/` のレンダリング周り）
- プリロードヒントの有無

#### CLS が高い場合
- 画像要素の width/height 指定状況
- フォント読み込み（FOUT/FOIT の発生）
- 動的コンテンツの挿入パターン

#### FCP が高い場合
- レンダーブロックリソース（CSS, 同期 JS）
- `<head>` 内のリソース読み込み順序
- サーバーレスポンス時間

### Step 4: 施策の作成

調査結果に基づき、具体的な施策を作成する。

各施策には以下を含めること：
- どのメトリクスにどれくらい効くか（定性的でよいが根拠を示す）
- 具体的な変更内容（ファイルパス、何をどう変えるか）
- 実装難易度の見積もり
- レギュレーション上の注意点があれば明記

### Step 5: proposals.md の出力

`perf-measurements/latest/proposals.md` に出力する：

```markdown
# 改善施策提案

**計測日時**: （report.md から転記）
**現在の推定スコア**: XX / 100 点（ページランディング）
**提案日時**: YYYY-MM-DD HH:MM

## 現状の課題サマリー

（失点分析から最も影響が大きい問題を2-3行で要約）

## 改善施策一覧

### 施策 1: [タイトル]

- **対象メトリクス**: TBT, LCP など
- **期待される改善**: 定性的な説明（例: TBT を大幅に削減）
- **優先度**: 高 / 中 / 低
- **実装難易度**: 低 / 中 / 高
- **影響範囲**: 全ページ / 特定ページ
- **具体的な作業内容**:
  1. ファイル `workspaces/xxx` の YYY を ZZZ に変更
  2. ...
- **対象ファイル**:
  - `workspaces/client/...`
  - `workspaces/app/...`
- **レギュレーション注意点**: （あれば）

### 施策 2: ...
（以下同様）

## 施策の優先順位

| 順位 | 施策 | 主な対象メトリクス | 期待改善 | 難易度 |
|------|------|--------------------|----------|--------|
| 1 | ... | TBT | 大 | 低 |
| 2 | ... | LCP | 中 | 中 |
| ... | | | | |

## 次のステップ

1. 施策 N から着手を推奨
2. 実装後は perf-measure で再計測して効果を確認
```

### Step 6: ユーザーへの報告

提案の要約をユーザーに伝える：
- 最も優先度の高い施策2-3個をピックアップ
- 実装に着手するか確認

## レギュレーション（施策提案時に必ず確認）

以下に違反する施策は絶対に提案しないこと：

1. **E2E テスト・VRT が通ること** - デザインやレイアウトを変えるような施策は VRT 差異3%以内であること
2. **Service Worker を register すること** - SW を削除する施策は NG
3. **漫画ページ画像の難読化必須** - 画像を平文で配信する施策は NG
4. **`POST /api/v1/initialize` で DB リセットできること** - DB 構造を壊す施策は NG

## このプロジェクトの既知の問題点

CLAUDE.md に記載されている意図的なパフォーマンス問題。施策のヒントとして活用する：

- 大量の不要 polyfill（core-js, es5/6/7-shim, regenerator-runtime）
- 重いライブラリ（three.js, canvaskit-wasm, jQuery, lodash, moment-timezone）
- 未最適化画像（14MB SVG ロゴ、大量の WOFF フォント）
- ミニファイ無効、ツリーシェイキング無効の tsup 設定
- 非効率な画像処理（jimp, image-js）
